# eslint-config builds every export through a single composer

The layer-ordering contract — re-apply the curated tunings last so they win, and apply `eslint-config-prettier` last to disable the presets' formatting rules — was hand-written into each entry point's config array, and had drifted: prettier landed mid-chain in `browser`, appeared twice in `react`, and was followed by `nextConfig` in `next` (so it was not the final layer) — all provably harmless, but the contract had no single owner. We invert the composition. Each concern becomes a raw, non-terminated **layer**, and a single internal composer `composeConfig(layers)` appends the same curated tail (`{ rules }`, then the TS-scoped `{ files: TS_FILES, rules: tsRules }`) and `eslint-config-prettier` exactly once, last. The tail is universal — there is no type-aware variant to pass: the type-aware tunings (`tsCheckedRules`) ride as their own TS-scoped layer inside `typescriptLayers`, beside the `projectService` that resolves them and disjoint from `tsRules`, so an export is type-aware purely by including those layers. Each `*Layers` export is an additive delta that does not include what it builds on; a terminal export spells out its ordered stack (`base` first) and **never spreads a terminated sibling export** — the practice that baked the tail and prettier into mid-array and let the contract erode. Rule data (`rules`, `tsRules`, `tsCheckedRules`, `unusedVarsOptions`) moves to `lib/tunings.js` and the composer to `lib/compose.js`; `base.js`, `typescript.js`, `browser.js`, and `react.js` become layer-providers (`base`, `typescriptLayers`, `browserEnvLayers`, `reactLayers`) rather than terminated configs others spread.

## Considered Options

- **A minimal tail helper** extracting just the shared tail used by `index.js` and `typescript.js`. Rejected: a half-measure. The tail is shared by only two of five exports, so a helper they call leaves the nesting trap intact — any layer added after a spread sibling's baked-in tail sits outside the curated/prettier protection — and does nothing for the `browser`/`react`/`next` prettier anomalies. It dedupes a symptom, not the cause.
- **Named semantic slots** (`composeConfig({ base, presets, env, tunings })`). Rejected: forces every layer into a taxonomy the real configs resist (is the import layer a preset? is testing-library env or preset? where does the react block sit?) and hides ordering more than a flat list — deepening the one genuine cost of the inversion (indirection) for no gain.
- **A `tsRules` variant parameter** on the composer (`composeConfig(layers, { tsRules })`), so type-aware exports pass the type-aware superset. Rejected: it duplicates one fact — "this export is type-aware" — across the layer list (include `typescriptLayers`, which sets `projectService`) and the call's options (pass the type-aware rules), with nothing enforcing they agree; mismatch is a hard parser crash. Folding the type-aware tunings into `typescriptLayers` as a disjoint scoped layer (they share no rule names with `tsRules`) co-locates them with the `projectService` they need, makes the tail universal, and lets type-awareness follow from the layers alone.
- **Leave it as-is.** Rejected: behaviour is provably correct — the resolved-config tests (ADR-0003) pin prettier-off and per-environment scoping across every export — but the contract still has no single owner, whether each export honours it is verified by eye, and it had already drifted once.

## Consequences

- "prettier exactly once, last" and "curated tunings win" become true by construction in every export. The composer is the sole place either can be expressed, so the Q-grilling fork over what "prettier last" means dissolves — the literal reading is now structural.
- The curated tail is variant-free: the composer always appends the same `{ rules }` + non-type-aware `{ tsRules }` + prettier, and an export becomes type-aware only by including `typescriptLayers` (which now carries `tsCheckedRules` beside its `projectService`). No call site restates "I am type-aware" twice, and the rule set is co-located with the parser option that resolves it.
- Removes the `tsRules` double-application the kernel carried: on the prior design `base.js` applied the TS-scoped tail and the entry points re-applied it. The composer is now the only applier; the kernel stops emitting the tail.
- Every `*Layers` export has one uniform meaning — an additive delta over the layers it builds on, never including them — so there is no per-binding "does this already contain the kernel?" question and no double-application footgun. Terminal exports compose the full stack explicitly (`base`, `typescriptLayers`, `browserEnvLayers`, `reactLayers`, …), which also makes each export's order legible at its own call site.
- Enables #109: the per-environment neutralization is one named `browserEnvLayers` entry in the list, ready for its own internal consolidation without reopening composition.
- Behaviour-preserving, asserted through the resolved config (ADR-0003): the resolved-config tests stay green, and a one-off full resolved-config diff (every export × fixture, before vs after, including rules, settings, globals, and parser options) confirms byte-identical output — so no changeset or release is owed (CONTEXT.md, _Release coverage_). Any non-empty diff would be consumer-facing and reverses that conclusion.
- The composer itself is never unit-tested in isolation (per ADR-0003 it is an ingredient); its behavioural contract is asserted through the resolved config. The one exception is "prettier exactly once, last", which the resolved config provably cannot distinguish — a duplicated or mid-chain prettier resolves formatting rules to off identically — so a structural test asserts it on each export's composed array, which also guards the "never spread a terminated sibling" convention.
- The cost is indirection — a reader must know the composer appends the curated tail and prettier, the one part of the order not visible at the call site (each export otherwise spells out its full stack). The "never spread a terminated sibling" rule, now backed by the structural prettier test, is the convention that keeps the inversion intact.

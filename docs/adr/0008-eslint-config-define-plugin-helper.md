# eslint-config registers single-block plugin wrappers through a definePlugin helper

`src/plugins/` held ~20 modules, each turning one upstream ESLint plugin into a flat-config block. For the shallow ones the registration shape — `files: [...DEFAULT_FILES]`, register the plugin under `plugins`, set `rules`, the `/** @type */` ceremony — was hand-rewritten per file, so changing it (adding a flat-config `name`, moving the default `files`) meant editing every file with no single owner. We introduce a tiny internal helper, `definePlugin({ name, plugin, rules, files?, settings?, ignores? })`, that concentrates that shape: `name` is the plugin's registration key (`'@next/next'`, `'better-tailwindcss'`, `'@eslint-community/eslint-comments'`), `files` defaults to `DEFAULT_FILES` when absent, takes an explicit array verbatim (`tailwindcss`), and **omits** the key entirely when `null` (the consumer-scoped `turbo`/`playwright`), and `settings`/`ignores` pass through (`compat`). The package stays file-per-plugin: each collapsed wrapper default-exports `definePlugin(...)`, so `base.js`'s import list and the `*Config` layer-binding convention (CONTEXT.md, _Layer_) are untouched.

The load-bearing decision is the **cut line**, and it is structural, not curation-depth: a wrapper uses `definePlugin` iff it is a single flat-config block registering a single plugin. That collapses 15 wrappers — `depend`, `no-secrets`, `no-unsanitized`, `no-use-extend-native`, `jsx-a11y`, `next`, `compat`, `tailwindcss`, `turbo`, `playwright`, `eslint-comments`, `promise`, `unicorn`, `sonarjs`, and `n` — including `n`, which carries the largest curated rule set of any single-block wrapper. It keeps hand-rolled bodies for the 6 that are structurally irreducible: `import` and `vitest` (multiple blocks, shared rule consts, resolver/`languageOptions`), `jsdoc` and `jsdoc-required` (a JS/TS flavor split across named exports), `graphql` (parser + processor), and `testing-library` (`dom`/`react` variants). The intuitive alternative — keep `n` (and the recommended-plus-overrides wrappers) hand-rolled so the file body advertises "this one is curated" — was rejected: it needs a per-plugin judgment on every wrapper, the verified-by-eye drift ADR-0007 was written to close, and a wrapper's depth is already visible in the size of its `rules` argument.

## Considered Options

- **An internal manifest array** listing the shallow plugins, mapped through the helper in `base.js`. Rejected: the public subpath exports `./plugins/{playwright,tailwindcss,turbo}` are shallow but must remain modules at those paths (changing the export map is consumer-facing), so a manifest can never own them — it forces split-brain, a manifest for the internal ones plus hand-files for the exported ones. It would also restructure `base.js`'s per-plugin imports. File-per-plugin is one uniform mechanism and leaves `base.js` and the `*Config` convention intact.
- **A depth-based cut line** — keep `n`, and the recommended-plus-overrides wrappers (`eslint-comments`, `promise`, `unicorn`, `sonarjs`), hand-rolled to signal curation. Rejected: "curated enough to hand-roll?" is a judgment call re-litigated per plugin and drifts by eye; the structural line ("one block, one plugin") is objective and self-applying. Depth stays legible in the `rules` argument, so a hand-rolled body signals nothing the rule set doesn't already show.
- **Widen `definePlugin` to absorb the deep keepers** — add `languageOptions`, `processor`, a multi-block array, plugin-less overlays — so all 21 plugins route through one factory. Rejected: the field set `{ name, plugin, rules, files?, settings?, ignores? }` is the exact envelope of the shallow shape, and the keepers spill outside it (`vitest` needs `languageOptions.globals` and a registration-less settings overlay; `graphql` a parser, a `processor`, and array exports; `import`'s `tsConfig` is a spread-derivation of `jsConfig`). Those extra needs _are_ the markers of depth, so widening the envelope erases the boundary it draws — the helper degrades into a pass-through whose parameters mirror the union of every plugin's needs, renaming flat config rather than hiding anything, and the deep files stop being the bespoke, self-narrating documents that earn them a file. The three keepers that _do_ fit the envelope (`testing-library`, `jsdoc`, `jsdoc-required`) already DRY their registration through a shared local `baseConfig`; routing them through the helper would restate `name`/`plugin`/`files` per sibling block, net longer.
- **Add flat-config `name`s in this change**, since the helper makes it a one-line addition. Rejected and split to #122: the helper owns only 15 of the package's dozens of composed blocks, so naming only those ships a half-named tree under `eslint --inspect-config`. Naming is coherent only as a comprehensive pass over every block (`base`, the composer tail, the keepers), and it adds a tooling-visible surface that would forfeit this change's no-changeset standing.
- **Leave it as-is.** Rejected: the registration shape stayed hand-written across 15 files with no owner, so changing it remained a 15-file edit — the same class of contract-without-an-owner ADR-0007 closed for layer ordering, still open for plugin registration.

## Consequences

- The single-plugin registration shape has one owner. Adding a flat-config `name`, or moving the default `files`, becomes a one-line change in `definePlugin` rather than 15 edits — the enabling move for #122.
- The cut line is structural: depth is read from the size of a wrapper's `rules` argument, never from whether its file hand-rolls the registration. A reader who finds `n` calling `definePlugin` while `import` does not learns the distinction is block-count, not curation.
- Behaviour-preserving, asserted through the resolved config (ADR-0003): a one-off diff (every standalone export × fixture via `calculateConfigForFile`, before vs after) plus a block-level `deepEqual` for the optional-export wrappers (`tailwindcss`, `turbo`, `playwright`) that sit in no standalone config confirms byte-identical output — so no changeset or release is owed (CONTEXT.md, _Release coverage_; precedent ADR-0007 / #121). Any non-empty diff reverses that conclusion.
- `definePlugin` is an ingredient, never unit-tested in isolation (ADR-0003); its contract is asserted through the resolved config. The one exception is the `files: null` → omit behaviour for the global wrappers (`turbo`, `playwright`): they belong to no standalone export, so `calculateConfigForFile` cannot see them, and a regression to `DEFAULT_FILES` would silently narrow them from global to default-files scope. A structural test asserts those blocks declare no `files` — the same guard ADR-0007 added for "prettier exactly once, last," the invariant resolved config provably cannot distinguish.
- The cost is indirection: a reader of a collapsed wrapper must know `definePlugin` supplies the `files` default and the `plugins` registration. The field set is exactly what the 15 wrappers need; a future single-block wrapper requiring `languageOptions` adds the field then, not speculatively.

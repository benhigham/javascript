# Configs target a modern, rolling browser/runtime floor

`@benhigham/browserslist-config` historically derived its browser floor from the `defaults` query (`['defaults and fully supports es6-module', ...last 2 major versions]`), which pulled in low-end and niche engines — QQ Browser 14.9, UC Browser 15.5, KaiOS, Chrome 109 — far below the interop level the rest of the toolchain assumes. That broad floor was the single root cause of a cluster of contradictions across the other configs, because the floor is the line every browser-compat rule measures against. We now derive the floor from one rolling, modern query — `'last 2 years and not dead and fully supports es6-module'`, intersected with `and` so the clauses _narrow_ each other (Browserslist combines comma- or array-separated queries with `or`, which would union them into a far broader set) — which drops the long tail and lands the floor at roughly Baseline "newly available" (today: Chrome 126 / Firefox 127 / Safari 17.6).

The broad floor put each config's browser-compat enforcement at war with its own modernization rules. In `stylelint-config`, `plugin/no-unsupported-browser-features` (caniuse data against the floor) flagged exactly the CSS the config tells you to write — `css-nesting` (required by `csstools/use-nesting`), `css-media-range-syntax` (required by `media-feature-range-notation`), plus `overflow: clip`, `:focus-visible`, and logical properties — while `plugin/use-baseline: newly` (Baseline data) considered all of those fine. Two browser-support gates, different data sources, opposite verdicts. In `eslint-config`, the floor forced two standing workarounds in the browser layer: `require-unicode-regexp` was widened from `requireFlag: 'v'` to `u` (the ES2024 `v` flag is a `SyntaxError` below Chrome 112, which SWC doesn't downlevel), and `unicorn/no-array-sort` was disabled (its only autofix, `.toSorted()`, was below the Chrome 110 floor).

We moved the floor rather than the alternatives — keeping the broad floor and peppering the configs with per-feature `ignore` lists, or dropping the browser-compat linters outright — because the configs' center of gravity is already modern (they bundle CSS nesting, range-notation, Baseline-newly, `.toSorted()`), and a floor that tracks Baseline is the honest expression of that. The trade-off is explicit: we drop support for old and niche browsers in exchange for modern CSS/JS that needs no downleveling workarounds. The floor is rolling, so it stays modern without manual bumps.

## Consequences

- **browserslist-config** ships the floor change as a **major** — consumers' build output targets narrow (the floor moves from `chrome 109` to the last two years of evergreen engines). The `./node` (maintained-Node) export is unchanged.
- **stylelint-config**: with the modern floor, `plugin/no-unsupported-browser-features` (plus `ignorePartialSupport: true` for the residual caniuse "partial support" pedantry) now agrees with `plugin/use-baseline: newly` instead of fighting it, so both stay on as complementary gates. `csstools/media-use-custom-media` is **removed entirely** (plugin + rule + dependency): `@custom-media` is not Baseline at any floor and the consuming Tailwind/PostCSS pipeline doesn't transpile it, so requiring it shipped dead `@media (--x)` rules — a runtime break, not just a lint contradiction. `csstools/use-nesting` and `media-feature-range-notation` stay. Ships as a minor (every change relaxes).
- **eslint-config**: the browser-layer floor-workaround block is deleted. The browser-only narrowing of `require-unicode-regexp` to `requireFlag: 'u'` is dropped — at the modern floor both `u` and `v` run, so the base's flag-agnostic `'error'` (require `u` _or_ `v`) applies uniformly. We deliberately do _not_ restore `requireFlag: 'v'`: the `v` flag is gated on the consumer's tsconfig `target` (ES2024), a compile-time axis the browserslist floor does not move, and `@benhigham/tsconfig` targets ES2023 — forcing `v` would re-require every consumer to set `target: ES2024` (the exact workaround the old `v` once forced). `unicorn/no-array-sort` comes back on; its `.toSorted()` autofix needs only `lib: ES2023`, which the tsconfig primitives already provide, so it carries no compile-time cost. Ships as a **major** (the `no-array-sort` re-enable surfaces new errors).
- The three packages are independent — none depends on `browserslist-config`. The distinction that matters: two gates read the consumer's resolved browserslist _dynamically_ at lint time — `compat/compat` (eslint) and `plugin/no-unsupported-browser-features` (stylelint) — so for those the floor change only bites once a consumer also adopts the new `browserslist-config`. Re-enabling `unicorn/no-array-sort`, and `plugin/use-baseline: newly`, instead encode a _static_ assumption that the floor is modern (unicorn rules and `use-baseline` don't read browserslist); they take effect the moment a consumer upgrades, regardless of their resolved browserslist. We therefore assume consumers pair these configs with `@benhigham/browserslist-config` (or an equally-modern floor), and coordinate the releases so the static assumption and the dynamic floor land in one go. A consumer who upgrades `eslint-config` alone, against an old floor, will get `.toSorted()` suggestions (from `no-array-sort`) that an old runtime may not support — the documented cost of the static model.

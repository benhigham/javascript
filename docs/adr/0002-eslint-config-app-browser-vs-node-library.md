# eslint-config distinguishes app/browser consumers from Node libraries

`@benhigham/eslint-config` historically applied library- and Node-oriented defaults to every consumer with no app-vs-library or browser-vs-Node distinction. Adopting the `/next` layer in a browser/static Next app surfaced ~900 ceremony errors and one build-breaking rule, all from that single root cause: rules that are correct for a published Node library are wrong for an application or for browser source. We now make that distinction explicit across four areas, so the base stays quality-focused and environment-specific assumptions live in the layer that owns the environment.

- **JSDoc is quality in the base, existence is opt-in.** The base composes the plugin's `contents` + `logical` + `stylistic` category presets (per the existing JS-flavor/TS split) — they validate JSDoc that exists without mandating it. The `requirements` category (`require-jsdoc`, `require-param`, `require-returns`, …) moves to a new opt-in export, `@benhigham/eslint-config/plugins/jsdoc-required`. Libraries documenting a public surface opt in with one line; apps don't pay for documentation ceremony on every symbol. The JS-flavor base keeps `jsdoc/no-types` off so JS files may carry JSDoc types (their only type source); TS files reject them — preserving the flavor split and keeping the opt-in's `require-*-type` rules satisfiable on JS.
- **`n` and `compat` are scoped to the environments they belong to.** In the browser/`next` layer, the Node-environment `n/prefer-global/*` family is neutralized on browser source — most sharply `n/prefer-global/process: never`, which flagged `process.env` reads and pushed a `node:process` import that breaks Next's build-time env replacement (the build-breaking rule). `compat` (a browser-API checker) is conversely scoped away from Node-environment files (config files, build scripts), where it produces false positives. Node-environment files in an app keep the full `n` rule set.
- **vitest builds up from `recommended`, not `configs.all`.** `configs.all` is a kitchen-sink that self-contradicts (`no-hooks` vs `require-hook`) and grows silently on every plugin minor bump. The base now layers a curated, high-value allowlist on top of `recommended` and deliberately omits high-ceremony/low-value rules (`no-hooks`, `require-test-timeout`, `max-expects`); `prefer-expect-assertions` is scoped to async/callback/loop tests where a missing assertion hides a real bug.
- **`import-x` allows devDependencies in test-support directories.** The devDependency allowlist now covers `test/`/`tests/`, `__mocks__/`, `__fixtures__/`, and `test-utils/` in addition to config and test files, so helpers/mocks/fixtures that import `vitest` or `@testing-library/*` but aren't named `*.{test,spec}.*` aren't flagged. The narrower vitest rule-scoping is unchanged — it stays on actual test files.

## Consequences

- Library consumers who relied on the base mandating JSDoc must now add `@benhigham/eslint-config/plugins/jsdoc-required` to keep that enforcement. No builds break from the move (rules only relax), and no existing export is removed, so it ships as a minor.
- Browser/`next` consumers can read `process.env` and other browser-safe globals without `n` fighting them, and `compat` no longer false-positives on their Node config files. The Node-library default (`.`, `./typescript`) is unchanged — `n` still applies everywhere there.
- The vitest rule set no longer drifts upward on plugin minor bumps; new vitest rules are adopted deliberately rather than automatically.
- JS-flavor consumers who type-check via JSDoc keep their inline types in the base (`no-types` stays off for JS), unlike a naive compose of the granular flavor presets, which enables `no-types` and would have erased those types.
